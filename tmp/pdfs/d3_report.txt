Project Overview

Deliverable 3 consolidates the project into a full campaign prototype built around four playable
districts: Town Center, Suburban Outskirts, Industrial District, and Research Complex.
The runtime campaign now spans 12 nights total (3 nights per level), with a final victory condition
that requires defeating Subject 23 in the final Research segment.
The gameplay loop remains intentionally consistent and readable:
 Day: follow objectives, scavenge resources, and prepare loadout/economy.
 Night: survive escalating waves with mutation and pacing pressure.
 Dawn: convert performance into progression via rewards, shopping, and upgrades.

Across Deliverable 3, this loop is no longer a two-level slice; it is implemented as a full route with
level identity, progression handoff, and a terminal encounter gate. The final branch requires both
night completion and boss elimination before the run can resolve as victory.
The report follows the deliverable rubric directly: Unity implementation and polish, art/aesthetics/sound,
narrative and systems integration, progression and balance, playtesting/iteration, and technical
implementation with submission readiness.

Each level has distinct geometry density, path logic, and color treatment:

routing decisions.
 Industrial (L3): tighter corridor pressure and denser obstacle routing.

3.2

UI Aesthetics and Readability

UI styling is centralized using shared theme tokens: dark background hierarchy, high-contrast text
tiers, and level accent colors mapped to campaign order (L1 green, L2 yellow-green, L3 amber, L4
red). This keeps menus, overlays, panels, and progression displays visually consistent while still
signaling stage identity.
Recent UI iteration in main branch focuses on readability and action clarity:
 fixed weapon-shop subtitle contrast and panel clipping,
 clearer guide text for interaction controls,
 simplified and more robust dialogue rendering,
 consistent campaign route communication in guide/menu overlays.

Two player-facing polish outcomes are especially important. First, after the locked-state button
visuals and subtitle contrast were fixed in the dawn shop, players were less likely to read unavailable
purchases as bugs and could more quickly infer unlock timing. Second, after Level 1 onboarding
hints were added for first combat contact, new players entered the night loop with less hesitation
and needed fewer facilitator prompts.

3.3

Audio Atmosphere and Feedback

Audio design combines procedural generation with runtime adaptive mixing:
 phase-aware music states (menu/day/night/dawn/boss),
 ambient layering and crossfades,

3

 dynamic tension model (base + transient spikes),
 voice ducking to keep comms legible during high-pressure moments,
 procedural SFX library (weapons, pickups, alarms, radio static, helicopter cues).

The system escalates intensity by context rather than static playback only. For example, night
tension increases by campaign depth, boss state enforces a higher tension floor, and communication
events trigger temporary ducking and threat peaks. This creates a more coherent sensory arc across
day exploration, combat escalation, and final-stage urgency.
A concrete audio iteration example came from combat readability feedback: earlier sessions showed
that some radio instructions were getting masked during high-action moments. The reactive mix
now ducks music beneath voice lines and then restores tension immediately after delivery, which
preserved urgency while improving instruction legibility during pressure.

Playtesting and Iteration

6.1

Runtime Verification

The repository includes PlayMode test suites that target gameplay-critical behavior:
 DaytimePrepRuntimeTests for prep rules, contested drops, and soft-penalty behavior,
 WeaponLoadoutRuntimeTests for four-slot loadout limits and rejection of overflow states,
 LevelTransitionResetRuntimeTests for inter-level reset correctness (points, upgrades, and
loadout),
 regression checks for loot and emergency-drop systems.

Recorded test artifacts in the repo show passing PlayMode runs for these runtime checks, used as
regression evidence while integrating campaign-level changes.
Manual playtesting used short repeated sessions (fresh Level 1 start, one full day–night–dawn loop,
then focused note-taking on confusion points). Each pass targeted one issue category before retesting.

Technical Implementation

The final prototype is organized around four cooperating runtime pillars:
 GameManager: campaign state, level/night indexing, progression transitions, and final victory
logic.
 GameFlowController: phase orchestration, day objectives, contested drops, and lifecycle
events.
 WaveManager: wave sequencing, enemy composition, adaptive scaling, and boss conditions.
 GameUI + objective systems: route communication, shop gating, and player-facing clarity.

District identity and pacing are data-driven through map-specific layout configs (landmarks, density, traversal, and per-night threat values), enabling targeted tuning without rewriting core loop
logic.

Conclusion

Deliverable 3 transitions Deadlight from a mid-project slice into a coherent full-campaign prototype.
The main branch now demonstrates complete route progression, integrated narrative-systems flow,
district-level visual/aesthetic identity, adaptive combat balance, and final-boss-gated completion.
Most importantly, the final prototype reflects iterative design rather than one-pass implementation: guidance, readability, pacing, and transition robustness were repeatedly refined to produce a
playable and assessable final build suitable for submission and showcase.
The strongest lesson from this phase is that readability is part of core design, not optional polish.
Short feedback loops (one observed confusion point, one targeted fix, one re-test pass) consistently
produced the largest gains in both system trust and player confidence.
For player experience, this meant a clearer run-level story: players can more quickly understand
what happened, why it happened, and what to do next. That clarity improved confidence in both
moment-to-moment decisions and long-range campaign planning, which was a central outcome goal
for this milestone.
